1) Sintesi tecnica (cosa c’è nelle fonti e come viene usato)
============================================================

A) Grover (Lezione 2)
- Ricerca in spazio non strutturato con complessità ~ √N.
- Esempio didattico: oracolo che marca uno stato (es. |1011⟩) via inversione di fase,
  poi diffusore (amplitude amplification) e misura.

B) Approccio “Hamiltonian-based” / simulazione (Lezione 1)
- Confronto gate-based vs analogico; definizione di Hamiltoniana e simulazione con circuiti.
- Esempio Qiskit: SparsePauliOp + PauliEvolutionGate per evoluzione (es. termine “ZZ”),
  con cenno a Trotterizzazione.
- Mappa concettuale citata: VQE, QAOA, quantum annealing (senza pipeline end-to-end).

C) QML / toolchain (Lezione 1–2 + qiskit.pdf + notebook)
- Building block presenti: zz_feature_map, efficient_su2, primitive Sampler/Estimator,
  NoiseModel con depolarizing_error, backend “fake” + transpiler (stima profondità/costi).
- Nota riproducibilità: nel materiale compaiono problemi di compatibilità (PennyLane/JAX)
  e dipendenze mancanti lato runtime.

============================================================
2) Analisi critica (limiti/semplificazioni del materiale)
============================================================

Grover
- Esempio “pulito” (stato marcato noto): nei problemi reali la parte difficile è costruire
  un oracolo utile e una funzione obiettivo non banale.
- Trade-off poco discussi: costo dell’oracolo, rumore/noise, scelta #iterazioni,
  robustezza della marcatura.

Hamiltoniane / simulazione
- Esempio minimale (pochi qubit, Hamiltoniana semplice): poco spazio a
  - errore di approssimazione (Trotter steps vs profondità)
  - metriche “end-to-end” (dalla simulazione al task di inferenza/apprendimento)

QML / laboratorio
- Ci sono i blocchi (feature map, ansatz, primitives, rumore), ma spesso manca la chiusura
  del cerchio su: task concreto, dataset/labeling, training/validazione, baseline classiche,
  benchmark sotto rumore.
- Stack non sempre stabile (fake provider/py deps, PennyLane+JAX): conviene scegliere una
  pipeline riproducibile (idealmente solo Qiskit + primitives del lab).

3) Spazio originale (direzione progettuale “coerente ma nuova”)
============================================================

Direzione: trasformare i “pezzi” del corso in una pipeline di identificazione di dinamiche
quantistiche (“fingerprinting” di Hamiltoniane).

Idea chiave (dati → modello)
- Simulazione Hamiltoniana per generare dati (Lezione 1)
- QML (feature map / variational circuit) per classificare o stimare parametri dinamici
- Grover non per “trovare |1011⟩”, ma per selezionare rapidamente il modello/parametro
  in un catalogo discreto di Hamiltoniane candidate

Differenza rispetto ai toy example
- Non: Grover su stringa + feature map senza task + evoluzione ZZ dimostrativa
- Sì: un task completo di inferenza + benchmark (anche con rumore)

============================================================
4) Proposta di progetto (scheda pronta)
============================================================

Titolo
Hamiltonian Fingerprinting: identificazione di modelli/parametri dinamici con QML + Grover

Algoritmi quantistici usati (visti a lezione)
- Simulazione Hamiltoniana con operatori di Pauli / PauliEvolutionGate (Lezione 1)
- QML con feature map (zz_feature_map) e/o ansatz efficient_su2 (lab)
- Grover / amplitude amplification per ricerca nel catalogo di modelli (Lezione 2)

Descrizione (target: 150–200 parole, testo “consegnabile”)
Obiettivo: dato un insieme di misure (o “tracce” di evoluzione) prodotte da un sistema
quantistico ignoto, identificare quale Hamiltoniana (o quale set discreto di parametri)
l’ha generato. Si definisce un catalogo di Hamiltoniane candidate (ad es. modelli Ising
su 2–4 qubit con coppie (J,h) discretizzate). Per ogni candidato si simula l’evoluzione
partendo da uno stato iniziale fissato e si costruisce un vettore di feature tramite
expectation value a tempi diversi. Su queste feature si addestra un modello QML (kernel
via feature map o classificatore variazionale) per riconoscere la classe/regime (ad es.
“campo alto vs basso” o l’ID del candidato). In parallelo, si implementa una versione
discreta di Grover: l’oracolo marca gli indici delle Hamiltoniane che spiegano le misure
entro una soglia (distanza sulle feature) e l’amplificazione aumenta la probabilità di
misurarle. Si valuta accuratezza e costo (profondità, shots) anche sotto rumore
depolarizzante.

Elemento di originalità (rispetto a slide/lab)
- Grover come motore di selezione modello (dati → Hamiltoniana), non ricerca di uno stato.
- Simulazione Hamiltoniana usata dentro un task di inferenza + benchmark sotto noise.

Problema concreto
System identification / model selection: riconoscere dinamiche (accoppiamenti/parametri)
da osservabili misurati; analogo a diagnostica di device o riconoscimento di “regimi”.

Approccio tecnico (pipeline)
1) Generazione dataset via evoluzione Hamiltoniana (PauliEvolutionGate; più tempi/feature)
2) QML: kernel (zz_feature_map) o VQC (efficient_su2) con training/validazione
3) Grover: indice candidati + oracolo “soglia” + diffusore; studio successo vs iterazioni/noise
4) Benchmark vs baseline classiche (grid search/random + SVM/RR su feature classiche)

Collegamento con il laboratorio
- Riusa: zz_feature_map, efficient_su2, Sampler/Estimator, NoiseModel+depolarizing_error
- (Opzionale) transpilation/backends “fake” per stimare profondità/costi

Output atteso
- Notebook di simulazione + mini-report con:
  - accuracy vs noise
  - query/iterazioni Grover vs probabilità di successo
  - profondità circuiti e shots → performance
  - confronto con baseline classiche

Fattibilità
- Simulabile con pochi qubit e cataloghi piccoli (8–32 candidati): oracolo implementabile
  in modo controllato e riproducibile con gli strumenti già presenti nel materiale.
